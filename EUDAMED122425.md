# 歐盟醫療器材法規 (MDR/IVDR) 數據治理與監管體系深度白皮書
## —— EUDAMED、UDI 邊境執法與全生命週期警戒系統之整合解析

### 執行摘要 (Executive Summary)

隨著 2025 年接近尾聲，歐盟醫療器材監管體系正迎來最具決定性的轉折點。歐盟醫療器材法規（MDR 2017/745）與體外診斷醫療器材法規（IVDR 2017/746）的實施，不僅是法律條文的更迭，更是一場以「數據」為核心的數位化監管革命。

本報告旨在全面剖析歐盟醫療器材資料庫（EUDAMED）的核心——UDI/裝置註冊模組的運作機制，探討 2025 年 6 月上線的 AIS/IMPORT PLUS 系統如何改變海關邊境執法，並深入論述 UDI 在不良事件通報與產品召回中的關鍵作用。對於全球製造商，特別是台灣的經濟營運者而言，理解並掌握這些數據流動的邏輯，是確保 2026 年強制合規期限後，產品能持續立足歐盟市場的關鍵。

### 第一章：歐盟醫療器材資料庫 (EUDAMED) 之單一識別系統模組
#### 1.1 EUDAMED 的建置背景與戰略意義

歐盟醫療器材資料庫（EUDAMED）的建立，是對過去歐盟醫療器材指令（MDD/AIMDD）時代監管漏洞的深刻反思。2010 年代的 PIP 乳房植入物醜聞暴露了國家數據庫碎片化、資訊不透明以及不良事件反應遲鈍等系統性風險。

EUDAMED 並非單純的資料儲存庫，它是歐盟構建「單一市場、單一數據視圖」的基石。其設計目的在於：
1.  **提升透明度**：讓公眾、醫療專業人員及監管機構能查詢器材資訊。
2.  **強化協調**：透過統一平台，使 27 個成員國的主管機關能即時共享資訊。
3.  **全生命週期追溯**：從產品註冊、臨床試驗、證書頒發到上市後監管，實現數據的無縫連結。

#### 1.2 核心樞紐：UDI/裝置註冊模組 (UDI/Device Registration Module)

在 EUDAMED 的六大模組中，**UDI/裝置註冊模組**扮演著「心臟」的角色。其他模組（如警戒、臨床調查、證書管理）都依賴此模組提供的識別數據來運作。

##### 1.2.1 核心識別碼：基本 UDI-DI (Basic UDI-DI, BUDI)
與美國 FDA GUDID 系統最大的不同在於，歐盟引入了 **Basic UDI-DI (BUDI)** 這一概念。
*   **定義**：BUDI 是一個高層級的、虛擬的識別符，它不會出現在產品實體的標籤或包裝上。
*   **功能**：它用於將具有相同預期用途、風險等級、基本設計和製造特性的「裝置家族」進行分組。
*   **戰略價值**：BUDI 是連接行政文件（如 CE 證書、符合性聲明 DoC、技術文件）與實體產品（UDI-DI）的橋樑。在不良事件報告中，BUDI 允許監管機構追蹤整個產品家族的系統性風險，而非僅限於單一型號。

##### 1.2.2 數據結構與核心要素
UDI/裝置註冊模組要求提交超過 100 個數據要素，其詳細程度遠超美國 FDA。這些數據可分為以下幾類：
*   **識別資訊**：BUDI、UDI-DI、參考型號。
*   **分類資訊**：風險等級（MDR Annex VIII）、GMDN 代碼（全球醫療器材命名）。
*   **臨床與安全**：預期用途、使用說明書（IFU）摘要、植入物卡資訊、臨床評估摘要（SSCP）。
*   **商業資訊**：上市狀態、銷售成員國。

| 項目 | 描述與細節 | 關鍵功能與 MDR/IVDR 關聯 |
| :--- | :--- | :--- |
| **核心識別碼** | **基本 UDI-DI (BUDI)** | 靜態識別符，用於連結證書與裝置家族，是 EUDAMED 模組間的數據樞紐。 |
| **法規基礎** | MDR (EU) 2017/745 Art. 29 | 取代舊有指令分散式管理，旨在提升追溯性與患者安全。 |

| **數據提交時機** | 上市前註冊 / 變更後一週內 | 確保數據的時效性。與 FDA 不同，歐盟要求部分數據需經公告機構驗證。 |

#### 1.3 實施進程與 2026 年強制期限

儘管 EUDAMED 經歷了多次延宕，但隨著 2025 年 11 月 26 日歐盟委員會正式發布 Decision (EU) 2025/2371，時間表已塵埃落定。
*   **現狀 (2025 年底)**：UDI/裝置註冊模組已處於自願使用階段，數百萬筆資料已錄入。
*   **轉折點 (2026 年 5 月)**：自 2026 年 5 月 28 日起，所有新產品及過渡期內的遺留裝置（Legacy Devices）必須在 EUDAMED 完成註冊，否則將被視為違規。
*   **全面上線 (2028 年)**：包含臨床與警戒模組在內的完整系統預計於 2028 年實現全功能整合。

#### 1.4 EUDAMED 與 FDA GUDID 的戰略比較

對於全球佈局的製造商，理解歐盟 EUDAMED 與美國 FDA GUDID 的差異至關重要。這不僅是 IT 欄位的對應，更是監管哲學的差異。

**表二：EUDAMED 與 FDA GUDID 深度比較分析**

| 比較項目 | **歐盟 EUDAMED UDI 模組** | **美國 FDA GUDID** | **全球製造商的應對策略** |

| **核心架構** | **多模組互聯**：以 BUDI 為樞紐，連結警戒、臨床、證書等六大模組。 | **單一資料庫**：主要作為 UDI 的儲存庫，與 MAUDE (不良事件) 系統相對獨立。 | 需建立主數據管理 (MDM) 系統，以支援不同區域的數據模型。 |

| **提交驗證** | **嚴格**：需先取得 SRN，部分高風險器材數據需公告機構審核。 | **自動化**：主要通過 FDA ESG 通道進行技術驗證。 | 歐盟的 SRN 申請與維護是額外的行政負擔，需提前規劃。 |

### 第二章：歐盟海關 UDI 審查要求與邊境執法
#### 2.1 從被動監管到主動攔截：AIS/IMPORT PLUS 系統

過去，歐盟的邊境監管主要依賴事後市場抽檢，缺乏類似美國海關與邊境保護局（CBP）的自動化攔截機制。然而，隨著 2025 年 6 月 19 日 **AIS/IMPORT PLUS** 系統的上線，這一局面徹底改變。

AIS/IMPORT PLUS 代表了歐盟海關作業的全面電子化。該系統將與歐盟單一窗口環境（EU Single Window Environment for Customs, EU-SW）整合，實現海關申報數據與 EUDAMED 數據的自動比對。

#### 2.2 進口商 (Importer) 的關鍵責任

在 MDR/IVDR 架構下，進口商不再只是物流角色，而是承擔法律責任的「經濟營運者」。MDR 第 16 條明確規定了進口商的驗證義務。由於歐盟海關不像美國那樣進行 100% 的 UDI 掃描，監管壓力轉嫁到了進口商身上。

**進口商必須執行的「自我驗證」程序：**
1.  **UDI 存在性驗證**：確認產品標籤上已標示正確的 UDI。
2.  **EUDAMED 比對**：確認該 UDI 已在 EUDAMED 中註冊，且狀態為「上市中」。
3.  **文件完整性**：確認 CE 證書與符合性聲明（DoC）有效且涵蓋該批次產品。
4.  **物理檢查**：抽樣檢查包裝完整性與標籤可讀性。

#### 2.3 電子申報與文件要求

為了配合 AIS/IMPORT PLUS 系統，所有進口申報文件必須結構化、數據化。傳統的紙本發票已無法滿足通關需求。

**表三：海關電子申報文件與格式要求**

| 文件類型 | 內容核心要求 | 電子格式與系統驗證機制 |
| :--- | :--- | :--- |
| **UDI 標籤數據** | 完整顯示 Basic UDI-DI + UDI-DI + PI (批號/效期)。 | **XML/JSON 格式**：需嵌入 AIDC (自動識別資料擷取) 條碼數據。海關系統透過 API 與 EUDAMED 比對。 |
| **EUDAMED 註冊證明** | 確認該裝置已完成註冊的官方憑證。 | **XML (MDCG 2019-4 標準)**：檔案命名規則為 `BasicUDI-DI_[GS1碼]_[版本]`，便於自動化讀取。 |
| **發碼機構聲明** | 證明使用的 UDI 代碼屬於授權機構 (如 GS1, HIBCC)。 | **年度 XML 更新**：需將企業的 GS1 前置碼與 EORI 號碼連結，一次申報，全年有效。 |
| **CE 證書 / DoC** | 證明產品符合 MDR/IVDR 法規。 | **結構化 PDF/XML**：需在文件中明確引用相關的 UDI-DI，並透過單一窗口 (PUESC) 上傳。 |
| **進口商識別** | 歐盟經濟營運者識別號 (EORI)。 | **EORI 號碼**：所有進口申報的強制性主鍵，需與 EUDAMED 中的 SRN 進行邏輯關聯。 |

#### 2.4 分階段強制與遺留裝置處理
雖然 AIS/IMPORT PLUS 於 2025 年上線，但針對不同風險等級的 UDI 查驗是分階段的：
*   **Class III / 植入式**：已於 2021 年全面強制，海關查驗最為嚴格。
*   **Class IIa / IIb**：2023 年起強制，目前是查驗重點。
*   **Class I / IVD Class B/C**：**2025 年 5 月 26 日**起強制。這意味著從現在開始，大量低風險器材將首次面臨 UDI 邊境檢查，這是台灣許多出口商面臨的最大挑戰。

### 第三章：UDI 在醫療器材回收與不良事件通報之應用
#### 3.1 警戒系統 (Vigilance System) 的數位化轉型

歐盟 MDR 第 87 條建立了一個嚴格的警戒報告系統。在舊指令時代，不良事件報告往往依賴模糊的產品描述（如「某品牌的心臟支架」），導致追溯困難。UDI 的引入，使得不良事件報告能精確定位到「單一特定型號（UDI-DI）」甚至「特定批次（UDI-PI）」。

#### 3.2 不良事件通報流程與時效

當發生不良事件時，製造商必須在 EUDAMED 的警戒模組中提交報告。BUDI-DI 在此時發揮關鍵作用，它將單一事件連結到整個產品家族的歷史記錄，幫助主管機關判斷這是否為系統性設計缺陷。

**通報時限要求：**
*   **嚴重公共衛生威脅**：獲知後 **2 天內**。
*   **死亡或健康嚴重惡化**：獲知後 **10 天內**。
*   **其他嚴重事故**：獲知後 **15 天內**。

#### 3.3 召回 (Recall) 與現場安全矯正措施 (FSCA)

UDI 系統最大的經濟效益在於「精準召回」。
*   **傳統模式**：若發現某一生產日期的產品有問題，往往因無法精確追蹤，需召回整個月甚至整季的產品，造成巨大的「過度召回（Over-recall）」損失。
*   **UDI 模式**：透過 **UDI-PI (生產標識)**，製造商可以精確鎖定受影響的特定序號或批號。在發布現場安全通知（FSN）時，明確列出受影響的 UDI 清單。

**表四：UDI 在不良事件與召回流程中的應用價值**

| 流程階段 | UDI 的具體應用 | 對製造商與主管機關的效益 |
| :--- | :--- | :--- |
| **事件識別** | 醫院透過掃描 UDI 記錄不良事件。 | 消除人工輸入錯誤，確保產品身分 100% 準確。 |
| **趨勢分析** | EUDAMED 透過 BUDI 聚合相似事件。 | 快速識別同一家族產品的系統性風險 (如特定材質疲勞)。 |
| **範圍界定** | 利用 UDI-PI (批號/序號) 鎖定問題產品。 | **減少召回範圍**：僅召回有問題的批次，大幅降低物流與賠償成本。 |
| **通知發布 (FSN)** | FSN 必須包含受影響的 UDI 清單。 | 讓經銷商與醫院能利用庫存管理系統自動篩查並隔離問題產品。 |
| **成效追蹤** | 追蹤已召回 UDI 的回收率。 | 即時掌握 FSCA 的執行進度，滿足主管機關的結案要求。 |

### 第四章：全球戰略分析與未來展望
#### 4.1 數據驅動的合規挑戰

對於製造商而言，EUDAMED 的強制實施帶來了前所未有的 IT 與營運挑戰：
1.  **數據維護負擔**：超過 100 個欄位的數據需保持最新。任何產品變更（如滅菌方法改變）都可能觸發新的 UDI-DI。
2.  **M2M 整合難度**：為了應對大量數據，企業需開發機器對機器（M2M）介面，這需要專業的 XML 編碼能力與 API 對接技術。
3.  **雙重合規成本**：同時滿足 FDA GUDID 與 EU EUDAMED 的要求，需要建立複雜的數據映射（Data Mapping）規則。

#### 4.2 2026-2030 未來展望

*   **AI 監管時代**：隨著 EUDAMED 數據的累積，歐盟將利用 AI 演算法進行自動化市場監測，預測供應鏈短缺風險，並主動偵測異常的不良事件趨勢。
*   **軟體即醫療器材 (SaMD)**：UDI 將更深入地應用於軟體版本控制，每一個微小的演算法更新都將有據可查。
*   **全球標準化**：歐盟與美國的 UDI 系統將透過 IMDRF 持續對齊，未來可能出現「全球單一存取點」的構想，降低跨國企業的合規成本。

**表五：歐盟 UDI 系統實施全景時間表與關鍵里程碑**

| 時間點 | 關鍵事件 | 對經濟營運者的影響與行動建議 |
| :--- | :--- | :--- |
| **2021/05** | Class III / 植入式 UDI 強制 | 高風險產品已完成合規，需持續維護數據準確性。 |
| **2025/05** | **Class I / IVD B/C UDI 強制** | 低風險產品（如手術器械、眼鏡）必須標示 UDI。**當前最緊迫任務**。 |
| **2025/06** | **AIS/IMPORT PLUS 上線** | 海關申報全面電子化，需確保 EORI 與 UDI 數據連結無誤。 |
| **2025/11** | EUDAMED 正式發布 (OJEU) | 觸發 6 個月的強制倒數計時。 |
| **2026/05** | **EUDAMED 核心模組強制使用** | 所有新舊產品必須完成 EUDAMED 註冊。未註冊者不得銷售。 |
| **2028** | EUDAMED 全功能上線 | 臨床與警戒模組完全整合，實現全生命週期數位監管。 |

### 結語
2025 年底的此刻，我們正站在歐盟醫療器材監管歷史的轉折點。EUDAMED 與 UDI 系統不再是遙遠的規劃，而是即將強制執行的現實。對於台灣及全球的醫療器材產業而言，這不僅是合規的挑戰，更是提升供應鏈透明度、優化產品品質管理的轉型契機。唯有擁抱數位化、精確掌握數據流，方能在這場監管變革中立於不敗之地。

2.  若遺留裝置（Legacy Device）計畫在 2027 年停產，是否仍需在 2026 年 5 月前完成 EUDAMED 註冊？
3.  針對「系統與程序包（System and Procedure Packs）」，其 UDI 註冊責任在於滅菌者還是組裝者？EUDAMED 對此有何特殊欄位要求？
4.  若產品的製造商位於台灣，但由歐盟授權代表（EC REP）協助註冊，若發生數據錯誤，法律責任如何歸屬？
5.  MDR 對於「可重複使用手術器械（Reusable Surgical Instruments）」的 UDI 直接標識（Direct Marking）有何特殊的實施時程與技術要求？

**EUDAMED 操作與技術**

7.  如何利用 Python 或其他工具將內部的 ERP 數據自動轉換為符合 EUDAMED 規範的 XML 格式？
8.  若 Basic UDI-DI 發生變更（例如設計重大變更），是否需要重新建立一條全新的 EUDAMED 記錄，還是可以進行版本更新？
